# 개발 일지 #4 — Supabase와 방치형 루프: 줍이가 기억을 갖게 된 날 🧠

> 2026-07-21 · 네 번째 기능 구현
> 이번 편에서 배우는 것: **BaaS(Supabase), 익명 인증과 쿠키 세션, Postgres 권한 2겹(grant + RLS), 스토어드 프로시저로 만드는 치트 불가능한 정산, zustand persist**

지금까지의 줍이에게는 슬픈 비밀이 있었습니다. **새로고침을 하면 모든 걸 잊어버린다**는 것.
열심히 모은 파편도, 쓰다듬어 올린 내구도도, F5 한 번이면 초기값으로 리셋됐습니다.
상태가 전부 자바스크립트 메모리(Zustand) 안에만 있었으니까요. 파이썬으로 치면
프로세스 재시작하면 사라지는 전역 변수에 게임 저장 데이터를 두고 있던 셈입니다.

펫 게임의 핵심은 **애착**이고, 애착은 **기억**에서 나옵니다. 그리고 우리 게임의 정체성인
방치형(Idle) 루프 — "자리를 비운 사이에도 줍이는 궤도를 돌며 일하고 있었다" — 는
서버가 시간을 기억해줘야만 가능합니다. 이번 편에서 이 두 가지를 한 번에 해결했습니다.

## 0. 전체 그림

```
브라우저                              서버                        Supabase (Postgres)
┌─────────────────┐    ┌──────────────────┐    ┌──────────────────────┐
│ Zustand 스토어    │    │ Server Actions    │    │ pets / profiles /     │
│  + persist       │◄──►│  bootPet()        │◄──►│ offline_logs          │
│ (localStorage)   │    │  syncPet()        │    │                       │
│                  │    │                   │    │ [RPC 함수]            │
│ PetLink 컴포넌트  │    │ (쿠키로 유저 식별) │    │  settle_offline() ★   │
│ (익명 로그인,     │    │                   │    │  sync_pet()           │
│  30초 동기화)     │    │                   │    │  ← 게임 규칙은 여기에  │
└─────────────────┘    └──────────────────┘    └──────────────────────┘
```

- **Supabase 키가 없으면?** 로컬 모드로 동작합니다. persist가 localStorage에 상태를
  지켜주니 새로고침에도 안전하지만, 서버 정산은 없습니다.
- **키가 있으면?** 접속할 때마다 서버가 "자리 비운 시간"을 정산해서 보상을 지급하고,
  플레이 중에는 30초마다 상태 스냅샷이 서버로 올라갑니다.

## 1. Supabase — 파이썬 개발자를 위한 한 줄 요약

Supabase는 **PostgreSQL + 인증 + 자동 REST API를 통째로 빌려 쓰는 서비스**입니다.
Django로 치면 모델(테이블)만 정의하면 admin과 DRF 엔드포인트가 공짜로 생기는 느낌인데,
한 발 더 나가서 **테이블을 만들면 REST API가 자동으로 생깁니다** (PostgREST라는 엔진이
`GET /rest/v1/pets?select=*` 같은 요청을 SQL로 번역해 줍니다).

"그럼 아무나 내 펫을 조회하고 수정할 수 있는 거 아냐?"라는 걱정이 바로 이번 편의
핵심 주제입니다. 답은 **DB 안의 권한 시스템**(3장)이 막아준다는 것.

로컬 개발은 Docker로 진짜 Supabase를 통째로 띄웁니다:

```bash
npx supabase start      # Postgres + Auth + REST 전부 로컬 기동
npx supabase db reset   # supabase/migrations/*.sql 순서대로 재적용
npx supabase stop       # 정리
```

마이그레이션 파일(`supabase/migrations/20260721000000_joops_init.sql`)이 스키마의
단일 진실입니다. 클라우드에 배포할 때도 같은 파일을 적용하면 됩니다 — Alembic/Django
migrations와 완전히 같은 사상입니다.

## 2. 익명 인증 — 가입 폼 없이 "한 명의 유저"가 되는 법

기획서의 온보딩 90초 룰: 첫 접속에서 텍스트 설명 없이 펫과 교감해야 합니다.
이메일 가입 폼은 그 자체로 실격이죠. 그래서 **익명 로그인**을 씁니다:

```ts
await supabase.auth.signInAnonymously();
```

이 한 줄이면 `auth.users`에 어엿한 유저 한 명이 생기고, 세션 토큰(JWT)이 발급됩니다.
나중에 이메일/소셜 계정을 **연결(link)** 하면 익명 시절의 펫이 그대로 승계됩니다.

재미있는 건 세션의 저장 위치입니다. `@supabase/ssr`의 클라이언트는 세션을
localStorage가 아니라 **쿠키**에 저장합니다. 왜냐?

- 쿠키는 **모든 HTTP 요청에 자동으로 실려 갑니다.**
- 그래서 브라우저에서 로그인하면, Server Action(서버)도 "지금 요청한 사람이 누구인지"
  압니다. Flask에서 `session["user_id"]`를 읽는 것과 같은 그림인데, 세션 저장소가
  서버 메모리가 아니라 서명된 JWT 쿠키라는 점이 다릅니다.
- 단, JWT는 1시간이면 만료됩니다. 갱신하려면 쿠키를 다시 써야 하는데 서버 컴포넌트는
  쿠키를 **읽기만** 할 수 있어서, 쿠키를 쓸 수 있는 유일한 길목인 `middleware.ts`가
  토큰 갱신을 전담합니다. (모든 라우트 앞에 걸린 WSGI 미들웨어라고 생각하면 됩니다.)

> **배울 포인트:** `lib/supabase/`에 클라이언트가 세 군데나 있는 이유 —
> 브라우저용(싱글톤), 서버용(요청마다 생성 — 그 요청의 쿠키를 읽어야 하니까),
> 미들웨어용(쿠키를 쓸 수 있는 유일한 곳). 실행 환경마다 쿠키를 다루는 능력이
> 다르기 때문에 생기는 분화입니다.

## 3. 치트가 불가능한 정산 — 게임 규칙을 DB 안에 넣기

방치형 게임의 두 가지 고전적인 치트:

1. **Time-cheat:** 폰 시계를 3일 뒤로 돌리고 접속 → "3일치 보상 주세요"
2. **API 치트:** 개발자 도구를 열고 `PATCH /rest/v1/pets` 로 `debris: 99999` 전송

둘 다 원천 차단했습니다. 핵심 아이디어는 **게임 규칙을 클라이언트도, Next.js 서버도
아닌 DB 함수(plpgsql) 안에 넣는 것**입니다.

### 3-1. 시간은 오직 DB의 now()

오프라인 정산 함수 `settle_offline()`은 인자를 하나도 받지 않습니다. 클라이언트가
"저 8시간 비웠어요"라고 주장할 통로 자체가 없습니다. 함수가 스스로 계산합니다:

```sql
v_away_sec := extract(epoch from (now() - v_pet.last_settled_at));
```

`now()`도 `last_settled_at`도 전부 DB가 기록한 시간입니다. 폰 시계는 등장하지 않습니다.

### 3-2. 배터리가 곧 보상 상한

방치형 게임은 보통 "오프라인 보상은 최대 12시간까지" 같은 인위적 캡을 둡니다.
우리는 그게 필요 없습니다 — **배터리(포만감)가 24시간이면 방전**되고, 방전되면 수집이
멈추니까요. 게임의 생존 지표가 그대로 경제 밸런스 장치가 되는, 개인적으로 아주
마음에 드는 설계입니다.

수집량 계산은 "데이터 용량이 가득 차면 효율 50%" 규칙 때문에 구간을 둘로 나눕니다:

```
활동 시간 = min(부재 시간, 배터리 / 시간당 소모율)      ← 배터리가 버틴 시간만 수집
1구간(효율 100%) = min(활동 시간, 데이터가 가득 차기까지 걸리는 시간)
2구간(효율 50%)  = 활동 시간 − 1구간
수집량 = 시간당 수집량 × (1구간 + 2구간 × 0.5)
```

적분 대신 구간 나누기 — 효율이 계단식으로 변하는 문제는 이렇게 구간별로 쪼개면
초등 산수가 됩니다. 상태 전이도 여기서 결정됩니다:
12시간 이상 방치 → **시무룩**, 방전 → **절전**, 방전 후 72시간 → **동면**. 사망은 없습니다.

### 3-3. 권한은 2겹이다: grant + RLS

Postgres 권한은 두 겹으로 걸립니다. 처음 배포했을 때 `permission denied for table pets`
에러를 만나며 몸으로 배운 내용입니다:

| 겹 | 질문 | 우리 설정 |
| --- | --- | --- |
| ① 테이블 grant | 이 역할이 이 테이블에 **무엇을** 할 수 있나 | `pets`: select, insert만. **update 없음!** |
| ② RLS 정책 | 허용된 동작을 **어느 행에서** 할 수 있나 | `auth.uid() = user_id` 인 행만 |

`pets`에 update grant를 **일부러 주지 않은 것**이 이번 편의 백미입니다. 개발자 도구로
`PATCH pets?debris=99999`를 쏘면 RLS까지 가기도 전에 권한 레벨에서 거절됩니다
(실제로 테스트했고, `42501 permission denied`가 돌아옵니다). 그럼 쓰기는 어떻게 하냐?

```sql
create function settle_offline() ... security definer;
```

`security definer`는 파이썬의 `sudo` 데코레이터쯤 됩니다 — 함수가 호출자 권한이 아니라
**함수 소유자 권한**으로 실행됩니다. 유저는 update 권한이 없지만, 검증 로직이 들어 있는
이 함수를 통해서만 쓰기가 일어납니다. "읽기는 열고, 쓰기는 함수로만"이라는 패턴입니다.

## 4. 이중 정산 버그를 미리 잡기 — last_settled_at의 의미론

플레이 중 30초마다 서버로 스냅샷을 올리는 `sync_pet()`에는 미묘하지만 중요한 한 줄이
있습니다:

```sql
last_settled_at = now()
```

이걸 빼면 이런 버그가 생깁니다: 10분 플레이하는 동안 클라이언트에서 배터리가 깎이고
→ 그 값이 서버에 저장되고 → 새로고침하면 정산 함수가 그 10분을 **또** 계산해서
배터리를 이중으로 깎습니다. `last_settled_at`의 의미를 "서버가 마지막으로 펫 상태를
확정한 시각"으로 정의하고, 확정하는 모든 경로(정산, 동기화)가 이 값을 당겨줘야
시간이 두 번 계산되지 않습니다.

> **배울 포인트:** 분산 시스템에서 타임스탬프 컬럼은 이름보다 **의미론**이 중요합니다.
> "마지막 로그인"인지 "마지막 정산"인지 "마지막 쓰기"인지에 따라 코드가 달라집니다.
> 우리는 기획서의 `last_login_at`(프로필용, 통계 목적)과 `last_settled_at`(정산 기준점)을
> 분리했습니다.

## 5. 클라이언트 쪽: persist와 관제 링크

### zustand persist — 상태 피클링

```ts
persist(store, {
  name: "joops.pet.v1",          // localStorage 키
  partialize: (s) => ({ ... }),  // 함수 빼고 데이터만 저장
})
```

파이썬의 pickle과 똑같은 고민입니다 — 전부 직렬화하면 안 되고(함수는 저장 불가),
스키마가 바뀌면 낡은 저장본과 충돌하니 키에 버전(`v1`)을 박습니다.

### PetLink — 화면 없는 컴포넌트

`components/pet-link.tsx`는 아무것도 그리지 않고 `return null` 하는 컴포넌트입니다.
React에서 이런 패턴은 "생명주기(마운트/언마운트)에 묶인 백그라운드 작업"을 컴포넌트
트리에 배치하는 방법입니다. 하는 일:

1. 세션 확보 (없으면 익명 가입)
2. `bootPet()` 호출 → 서버 정산 결과로 스토어 덮어쓰기 → 보고할 게 있으면 귀환 보고 모달
3. 스토어 구독 → **바뀌었을 때만** 30초마다 `syncPet()` (탭 연타마다 서버를 부르지 않기)
4. 탭을 벗어나는 순간(`visibilitychange`) 마지막 스냅샷 밀어 넣기

동기화 실패 시엔 dirty 플래그를 되살려 다음 주기에 재시도합니다. 30초 안의 플레이는
유실될 수 있지만, 정산 기준점(`last_settled_at`)이 마지막 동기화 시점이라 유저가
손해 보는 시간은 최대 30초 — 수용 가능한 트레이드오프로 판단했습니다.

## 7. 정직한 한계와 다음 단계

- `sync_pet()`은 아직 클라이언트 스냅샷을 **범위 클램프만 하고 믿어줍니다.**
  "30초 만에 파편 +500은 물리적으로 불가능" 같은 속도 검증은 다음 과제입니다.
- 홈 화면이 아직 시무룩/동면 무드를 표현하지 못합니다 (모달에서만 알려줌).
- 푸시 구독을 DB에 저장하고, 펫 상태를 보고 알아서 알림을 보내는 cron(Edge Function)이
  자연스러운 다음 편입니다 — "배터리가 3%예요, 줍이가 기다려요" 같은 알림이요.
- 클라우드 전환: Supabase 프로젝트 만들기 → **Authentication에서 Anonymous sign-ins
  켜기** → 마이그레이션 적용 → Vercel 환경변수 교체. 자세한 순서는 `supabase/README.md`.

> **덧붙임 (같은 날):** 클라우드 프로젝트에 예전 실험용 `pets` 테이블이 이미 살고
> 있다는 게 발견되어, 우리 DB 객체 전부에 `joop_01_` 접두사를 붙였습니다
> (`joop_01_pets`, `joop_01_settle_offline()` …). 스키마 없는 Postgres에서 접두사는
> 가난한 자의 네임스페이스지만, PostgREST 설정을 건드리지 않고 한 프로젝트에
> 여러 실험을 공존시키는 가장 실용적인 방법입니다. 본문 코드의 이름은 개명 전
> 기준이니 현재 이름은 마이그레이션 파일을 보세요.
