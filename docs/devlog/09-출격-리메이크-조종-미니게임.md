> 이번 편에서 배우는 것: **명령형 DOM 게임 루프(React의 escape hatch), 오일러 적분과 관성 물리, 가중치 랜덤, '게임 필'을 만드는 비대칭 판정, 조작감/난이도 상수 분리**

7편에서 만든 액션 모드(이모지 타임어택)를 통째로 리메이크했습니다. 이유는 두 가지:

1. **그래픽** — 이모지 ☄️🪨🌟 대신, 에셋 팩의 카와이 스티커 SVG로. 이제 출격
   화면의 줍이가 홈 화면 3D 줍이와 같은 얼굴입니다 (진화 단계·장비까지 반영!).
2. **게임 방식** — 같은 조직의 형제 프로젝트 **jd-03(STELLAPET)** 의 '조종'
   미니게임 방식을 이식했습니다. 파편이 떨어지는 걸 받아치는 게임에서,
   **줍이를 직접 조종해 파편 사이를 날아다니는** 게임으로.

> jd-03의 미니게임은 jd-02에서 한 번 검증된 루프를 또 이식한 것이라,
> 우리는 "세 번째 이식"인 셈입니다. 검증된 재미는 베끼는 게 맞습니다.

## 1. 새 게임 방식: 연료 서바이벌

45초 타이머는 버렸습니다. 대신 **추진 에너지**(라운드 전용 게이지 100)가
이 게임의 시계입니다:

- **조그셔틀 조종** — 화면 아무 곳이나 누르면 그 자리에 가상 스틱이 생기고,
  드래그한 방향으로 분사합니다. 드래그 깊이가 액셀 페달: 얕게 끌면 1단(찔끔),
  깊게 끌면 3단(콰앙). 세게 밟을수록 에너지도 세게 답니다 — `[2, 6, 14]`/초.
- **파편 수거** — 소형(칩·볼트·너트 +1) / 중형(기어·안테나 +3) / 대형(연료탱크
  +8, 느리고 큼직한 대어). 닿으면 "냠!" 하고 빨려 들어옵니다.
- **고속 파편(빨간 것)** — 맞으면 에너지 -15, 정산 때 내구도 -5. 1.2초 무적과
  화면 흔들림으로 "맞았다"를 몸으로 알려줍니다.
- **에너지 소진 → 표류 4초 → 자동 귀환.** 그런데 표류 중 반짝이는 **태양전지
  파편**을 먹으면 **"재점화!"** — 죽었다 살아나는 이 순간이 게임의 명장면입니다.
- 출격 전 **위험 수당제** 브리핑: 안전 ×1.0 / 표준 ×1.5 / 밀집 ×2.0. 위험한
  구역일수록 고속 파편이 잦고 빠릅니다. (기획서 V3의 시스템 — 에셋 팩에
  육각 배지가 이미 준비되어 있었어요. 원칙대로 LEO 같은 물리 용어는 숨기고
  '밀집 구역' 같은 친숙한 이름만 노출합니다.)

본편 육성도 조종감에 스며듭니다: AI 코어 레벨당 추진 가속 +6%, 3단계 장비별
패시브(자석 = 끌어당김 범위 +60%, 그물 = 획득 판정 +8px, 레이저 = 피격 손실
절반 + 무적 +50%). 열심히 키운 줍이는 실제로 더 잘 납니다.

## 2. 명령형 DOM 게임 루프 — React 안에 pygame 숨기기

7편에서는 "진실은 ref에, 매 프레임 `setTick`으로 리렌더"였습니다. 이번엔 한 발
더 나가서, **매 프레임 리렌더 자체를 없앴습니다**. 게임 루프가 DOM을 직접
만집니다:

```ts
// 스폰: rAF 루프가 <img>를 직접 만들어 붙이고
const el = document.createElement("img");
el.src = DEBRIS_SRC[kind];
layer.appendChild(el);

// 매 프레임: transform만 갈아끼우고
el.style.transform = `translate3d(${x}px, ${y}px, 0) rotate(${rot}deg)`;

// 소멸: 직접 뗀다
el.remove();
```

파이썬으로 치면 **pygame의 메인 루프를 React 컴포넌트 안에 숨긴 것**입니다.
React는 껍데기(브리핑·HUD·결과 모달)만 선언적으로 그리고, 초당 60번 변하는
것들은 루프가 명령형으로 처리합니다. 이런 걸 React의 **escape hatch**(비상구)
라고 불러요 — 공식 문서도 "게임처럼 극단적으로 자주 변하는 상태는 React
바깥에 두라"고 안내합니다.

HUD 숫자(에너지·수거량·시간)는 매 프레임 `setState`하지만 값이 정수라 대부분
그대로예요 — React는 `Object.is`로 같은 값이면 리렌더를 건너뜁니다. 결과적으로
리렌더는 초당 몇 번 수준.

**왜 Canvas가 아니라 DOM인가?** jd-03은 Canvas 2D 픽셀아트였지만 우리는:

- 에셋이 **벡터 SVG**입니다. `<img>`로 띄우면 어떤 크기·DPR에서도 공짜로
  선명해요. Canvas에 SVG를 그리려면 해상도별 래스터 버퍼를 직접 관리해야 합니다.
- Canvas는 매 프레임 전체 화면을 다시 칠하지만, 합성 레이어의
  `translate3d` 이동은 **리페인트가 없습니다** — GPU가 이미 그려둔 층을
  옮기기만 해요. 엔티티 ~40개면 DOM이 오히려 싸고, 모바일 발열 원칙(개발
  원칙 3)에도 맞습니다.

## 3. 관성 물리 — 위치가 아니라 속도를 조종한다

기존 게임은 손가락 위치로 줍이를 **순간이동**시켰습니다. 새 게임은 손가락이
**힘(가속도)** 을 줍니다:

```ts
vx += (dx / dist) * acc * dt; // 가속도를 속도에 누적하고
vy += (dy / dist) * acc * dt;
pet.x += vx * dt;             // 속도를 위치에 누적한다
pet.y += vy * dt;
```

수치해석 시간에 본 **오일러 적분**(Euler integration) 그대로입니다 —
`v += a·dt; x += v·dt`. `(dx/dist, dy/dist)`는 드래그 방향의 **단위벡터**고요.
여기에 세 가지 양념이 우주 조종감을 만듭니다:

```ts
vx -= vx * friction * dt;   // ① 마찰 감쇠 — 서서히 잦아들고
if (sp < minSpeed) { ... }  // ② 최소 표류 속도 — 완전히 멈추진 않고
vx *= -bounce;              // ③ 벽 반동 — 화면 밖 대신 통통
```

손을 떼도 줍이는 관성으로 미끄러집니다. "멈추고 싶은데 못 멈추는" 답답함이
아니라 "우주니까 당연하지"가 되는 게 이 조작의 재미 포인트예요.

`dt = min(0.05, 실제 경과)`의 상한도 여전히 중요합니다. 탭이 백그라운드에
갔다 오면 dt가 수십 초일 수 있는데, 그대로 곱하면 줍이가 벽을 **뚫고**
화면 밖 우주로 사출됩니다(터널링). 상한이 그걸 막아요.

## 4. 가중치 랜덤 — random.choices의 손구현

파편 종류는 균등 랜덤이 아닙니다. 칩은 흔하고 연료탱크는 귀합니다:

```ts
// 파이썬: random.choices(kinds, weights=[30, 16, 14, ...])[0]
const totalW = table.reduce((acc, k) => acc + k.w, 0);
let r = Math.random() * totalW;
for (const k of table) {
  r -= k.w;
  if (r <= 0) return k.kind;
}
```

파이썬의 `random.choices(weights=...)`를 JS에는 내장이 없어서 손으로 만든
겁니다. 룰렛을 상상하면 돼요 — 가중치만큼 넓은 칸들을 붙여 놓고 공을 굴리는
것. 스폰 간격에도 `±30% 지터`를 줘서 리듬이 외워지지 않게 합니다.

## 5. '게임 필'의 비밀 — 후하게 먹고, 짜게 맞기

jd-03에서 가장 배울 만했던 부분입니다. 충돌 판정이 **비대칭**이에요:

```ts
// 획득: 펫 반지름 + 파편 반지름 + 8px 보너스 — 보이는 것보다 후하게
dist < petR + j.radius + eatBonus
// 피격: 펫 반지름을 0.75배로 줄여서 — 보이는 것보다 짜게
dist < petR * hitShrink + j.radius
```

플레이어는 "아깝게 놓쳤다"에는 분노하고 "스쳤는데 맞았다"에는 더 분노합니다.
양쪽 다 플레이어에게 유리하게 속이면 게임이 "손에 붙는다"고 느껴요. 여기에
**자석**(위험물 빼고 근처 파편을 슬쩍 끌어오기)까지 더하면, 플레이어는 자기가
잘한다고 착각합니다 — 그 착각이 재미입니다.

## 6. TUNE vs DIFFICULTY — 상수를 두 서랍에 나눈 이유

`sortie-tuning.ts`의 상수는 두 묶음입니다:

| 서랍 | 내용 | 언제 만지나 |
| --- | --- | --- |
| `TUNE` | 스틱 반경, 가속, 감쇠, 판정 보너스… **조작감** | 게임의 성격을 바꿀 때 (신중히) |
| `DIFFICULTY` | 시작 에너지, 소모율, 위험물 빈도… **난이도** | 밸런스 패치 때 (부담 없이) |

"이 게임 너무 어려워요"라는 피드백에 조작감까지 흔들면 손맛을 잃습니다.
난이도 서랍만 열게 강제하는 구조적 장치예요. 파이썬으로 치면 설정을
`config.py`와 `balance.yaml`로 나누는 감각입니다. (jd-03는 여기에 환경변수
오버라이드까지 얹었는데, 우리는 다음 과제로 남겼습니다.)

## 7. 오늘의 버그: 포인터 캡처가 버튼을 삼켰다

"화면 아무 곳이나 누르면 스틱"을 위해 `field.setPointerCapture(pointerId)`를
쓰는데, 이게 **이후 포인터 이벤트를 전부 필드로 리다이렉트**합니다. 문제는
클릭 이벤트까지 — [귀환] 버튼을 눌러도 클릭이 필드에게 가로채여 버튼이 안
눌렸어요. Playwright 스모크 테스트(배너 → 브리핑 → 비행 → 정산)에서 발견.

```ts
// 버튼 위에서 시작한 포인터는 스틱을 만들지 않는다
if (e.target instanceof Element && e.target.closest("button")) return;
```

교훈: **전체 화면을 제스처 영역으로 쓰는 UI에선, 그 위의 버튼들이 캡처에
먹히지 않는지 반드시 실기기/자동화로 확인할 것.**

## 8. 배선은 그대로 — 계약이 있으면 리메이크가 싸다

게임을 통째로 갈아엎었는데 스토어와 서버는 **한 줄도 안 바뀌었습니다**.
비결은 8편까지 지켜온 정산 계약:

```ts
finishSortie({ debris, exp, durabilityLoss }); // 이 서명만 지키면
```

진입(돌발 배너 → `startSortie()` 배터리 -10)도, 정산 후 30초 동기화가 서버로
나르는 것도, 데이터 축적(`dataUsed += debris × 화물칸 압축`)과 내구도 하한
30도 전부 기존 파이프라인이 처리합니다. 인터페이스를 좁게 유지하면 구현은
마음껏 갈아치울 수 있다 — 이번 리메이크의 가장 실용적인 교훈입니다.
