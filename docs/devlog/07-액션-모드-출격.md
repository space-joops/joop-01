## 1. 제스처 3종이 곧 전투 조작

기획서의 제스처 매핑을 그대로 전투로 옮겼습니다. 홈 화면 돌봄과 같은 손짓이라
새 조작을 배울 필요가 없습니다 (첫 3초 안내도 "☄️ 탭! · 🪨 끌어서 피하기! ·
🌟 꾹 눌렀다 떼기!"가 전부):

| 제스처 | 돌봄 모드 | 액션 모드 |
| --- | --- | --- |
| Tap | 파편 냠냠 | ☄️ 레이저 요격 (+1) |
| Drag | 쓰다듬기 수리 | 🪨 암석 회피 (피격 시 내구도 -5, 하한 30) |
| Hold & Release | (예정: 기지개) | 🌟 골드 코어 조준·로봇 팔 포획 (+10, EXP 15) |

암석은 **줍이의 현재 위치를 노리고** 떨어집니다 — 가만히 있으면 맞습니다.
회피를 강제하는 가장 값싼 난이도 장치예요. 시간이 갈수록 스폰이 1.5배까지
빨라져 후반 긴장감을 만듭니다.

## 2. rAF 게임 루프 — setInterval이 아닌 이유

게임 루프는 `requestAnimationFrame`(rAF)으로 돌립니다:

```ts
const loop = () => {
  raf = requestAnimationFrame(loop);
  const dt = Math.min(now - prev, 0.05); // 프레임 드랍 시 폭주 방지 캡
  entity.y += entity.vy * dt;            // 속도 × 시간 — 프레임레이트 독립
};
```

- rAF는 **화면 주사율에 동기화**됩니다(60Hz 폰이면 초당 60번). setInterval은
  브라우저 렌더링과 어긋나 뚝뚝 끊깁니다.
- 탭이 백그라운드로 가면 rAF는 자동으로 멈춥니다 — 배터리 절약 공짜 획득.
- `dt` 곱셈은 5편(지수 감쇠)에서 배운 "프레임 수를 믿지 마라" 원칙의 재활용.
  120Hz 폰에서도 파편이 두 배로 빨라지지 않습니다.

## 3. ref 게임 상태 + 틱 리렌더 — React와 게임 루프의 동거

초당 60번 setState로 엔티티 배열을 갱신하면 어떨까요? 되긴 하지만, 매 프레임
배열 복사 + 리액트 조정(reconciliation) 비용이 저사양 폰에서 부담입니다.
그래서 분업했습니다:

```ts
const entitiesRef = useRef<Entity[]>([]);  // 진실은 ref에 (루프가 직접 mutate)
const [, setTick] = useState(0);           // 프레임 끝에 +1 — "다시 그려줘" 신호만
```

5편의 useFrame 패턴과 같은 사상입니다 — **게임 데이터와 렌더 트리거의 분리**.
파이썬으로 치면 시뮬레이션 상태는 numpy 배열에 두고, 시각화 갱신 신호만
이벤트로 쏘는 구조랑 비슷해요.

## 4. 제스처 판별의 우선순위 설계

한 손가락이 세 가지 의미를 가지려면 **판별 순서**가 필요합니다:

1. ☄️ 위에서 눌렀다? → 엔티티 자신의 `onPointerDown`이 먹고
   `stopPropagation()` — 컨테이너까지 버블링되면 회피 기동으로 오판된다.
   (이벤트 버블링: 자식의 이벤트가 부모로 올라가는 DOM의 기본 동작)
2. 누른 채 8px 이상 움직였다? → **회피 기동** (조준 게이지 리셋).
3. 움직이지 않고 버텼다? → **조준** — 게이지가 1초에 걸쳐 차오르고,
   55% 이상에서 떼면 가장 가까운 골드 코어를 로봇 팔로 포획.

작은 파편의 히트박스에 `p-2`(패딩)를 준 것도 포인트 — 이모지 자체는 작지만
엄지 판정은 후하게. 모바일 게임 UX의 오래된 지혜입니다.

## 5. 정산은 한 번에 — finishSortie

라운드 중 보상을 실시간으로 스토어에 넣지 않고, 통계만 쌓다가 귀환 시
`finishSortie()`로 **한 번에** 반영합니다. 파편·EXP·데이터 증가와 내구도
하한(30) 처리가 한 곳에서 끝나고, 이미 있던 30초 동기화가 서버로 나릅니다 —
4편에서 만든 파이프라인에 그대로 올라타는 거죠. 새 서버 코드 0줄.
