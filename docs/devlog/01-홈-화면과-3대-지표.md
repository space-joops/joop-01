## 1. 서버 컴포넌트 vs 클라이언트 컴포넌트 — 왜 파일 첫 줄에 "use client"를 쓸까?

Next.js App Router의 컴포넌트는 기본적으로 **서버 컴포넌트**입니다. 파이썬으로 비유하면
Flask에서 Jinja2 템플릿을 렌더링해 완성된 HTML을 내려보내는 것과 같아요. 서버에서 한 번
그려지고 끝이라, 브라우저에서 클릭에 반응하거나 상태를 가질 수 없습니다.

반면 파일 맨 위에 `"use client"`를 선언한 **클라이언트 컴포넌트**는 자바스크립트 코드가
브라우저로 전송되어 그곳에서 살아 움직입니다. `useState`, `onClick`, `setInterval` 같은
"상호작용"은 전부 여기서만 가능합니다.

우리 게임의 분업:

- `app/page.tsx` — 서버 컴포넌트로 남겨뒀습니다. 나중에 **Supabase에서 유저/펫 데이터를
  읽어오는 곳**이 될 예정이라서요. (DB 접근은 서버에서 하는 게 안전하니까요. Flask 뷰 함수에서
  DB 조회 후 템플릿에 넘기는 그림과 같습니다.)
- `components/home/*` — 전부 클라이언트 컴포넌트. 터치와 애니메이션 덩어리니까요.

> **배울 포인트:** "일단 전부 클라이언트로" 하지 않고, 상호작용이 필요한 최소 단위만
> 클라이언트로 만드는 게 App Router의 정석입니다. 서버 컴포넌트는 JS 번들에 포함되지
> 않아서 페이지가 가벼워집니다.

## 2. Zustand — 전역 싱글톤 dataclass + 옵저버 패턴

`stores/pet-store.ts`가 게임의 두뇌입니다. 파이썬 감각으로 읽으면:

```python
# 파이썬으로 치면 대략 이런 느낌
@dataclass
class PetState:
    battery: int = 80
    durability: int = 65
    data_used: int = 40

    def eat_debris(self):
        if self.battery <= 0:  # 절전 모드면 못 먹음
            return
        self.data_used = min(100, self.data_used + 6)
        ...

pet = PetState()  # 전역 싱글톤
```

차이점은 **구독(subscribe)** 입니다. React 컴포넌트가 `usePetStore((s) => s.battery)`처럼
"나는 배터리만 볼래요"라고 등록해 두면, 배터리가 바뀔 때 그 컴포넌트만 자동으로 다시
렌더링됩니다. 옵저버 패턴이 프레임워크에 녹아 있는 셈이죠.

여기서 중요한 습관 하나:

```tsx
// 좋음: 값 하나씩 골라 구독 → 배터리가 변해도 내구도 게이지는 안 다시 그림
const battery = usePetStore((s) => s.battery);

// 나쁨: 스토어 통째로 구독 → 아무 값이나 바뀌면 전부 재렌더링
const store = usePetStore();
```

그리고 상태 갱신은 **직접 수정이 아니라 새 객체 반환**입니다. 파이썬처럼
`state.battery -= 2`라고 쓰지 않고 `set((state) => ({ battery: state.battery - 2 }))`로
"새 값이 담긴 딕셔너리"를 돌려줍니다. React 세계는 불변성(immutability)을 기준으로
"뭐가 바뀌었는지"를 감지하기 때문입니다.

### 파생 상태는 저장하지 않는다

`isSleeping`(배터리 0 = 절전)이나 `isDataFull`은 스토어에 저장하지 않고 함수로 계산합니다.
`sleeping`이라는 값을 따로 저장하면 배터리와 어긋나는 순간(버그)이 생길 수 있어요.
**"원본은 하나만 저장하고 나머지는 계산한다"** — DB 정규화와 같은 원리입니다.

## 3. 제스처 판별 — 탭과 드래그는 어떻게 구분할까?

브라우저에 "탭 이벤트"나 "쓰다듬기 이벤트"는 없습니다. 있는 건 `pointerdown` /
`pointermove` / `pointerup`뿐이라, 기획서의 제스처는 우리가 직접 판별해야 합니다.

`pet-satellite.tsx`의 규칙:

| 판정 | 조건 | 결과 |
| --- | --- | --- |
| **Tap** | 총 이동 ≤ 10px, 누른 시간 ≤ 300ms | 파편 냠냠 ✨ |
| **Drag** | 이동 > 10px | 누적 28px마다 쓰다듬기 1회 💚 |

구현 디테일 세 가지:

1. **`setPointerCapture`** — 문지르다 손가락이 영역 밖으로 나가도 이벤트를 계속 받게 하는
   장치입니다. 이게 없으면 쓰다듬다가 뚝 끊깁니다.
2. **`useRef`로 추적** — 누른 시각, 누적 거리 같은 값은 화면에 표시되지 않으므로
   `useState` 대신 `useRef`에 담습니다. `useState`는 값이 바뀔 때마다 재렌더링을 일으키는데,
   `pointermove`는 초당 수십 번 발생하니 ref가 아니면 성능이 무너집니다.
   (ref = "렌더링과 무관한 인스턴스 변수", 파이썬 클래스의 `self._tracking` 같은 것)
3. **CSS `touch-action: none`** — 이걸 빼면 드래그할 때 브라우저가 "스크롤이구나!" 하고
   가로채 갑니다. "이 영역의 터치는 내가 처리할게"라는 선언입니다.

## 4. Hydration — 별 배경에 Math.random()을 못 쓴 이유

배경의 별 28개를 랜덤 위치에 뿌리고 싶었는데, 렌더링 코드에서 `Math.random()`을 쓰면
안 됩니다. Next.js는 서버에서 HTML을 먼저 그리고, 브라우저가 같은 컴포넌트를 다시 실행해
"이어받기(hydration)"를 하는데, 서버와 브라우저의 랜덤값이 다르면 두 결과물이 어긋나서
hydration 오류가 납니다.

그래서 **인덱스 기반 의사 랜덤 공식**으로 "항상 같은 랜덤"을 만들었습니다:

```tsx
const STARS = Array.from({ length: 28 }, (_, i) => ({
  left: (i * 37 + 11) % 100,  // 소수(prime) 곱셈 → 규칙 없어 보이는 분포
  top: (i * 53 + 17) % 100,
  ...
}));
```

서버가 계산해도, 브라우저가 계산해도 결과가 같으니 안전합니다. 파이썬의
`random.seed(42)`로 재현 가능한 랜덤을 만드는 것과 같은 발상이에요.

## 5. 모바일 게임 CSS 방어 장비 4종

`globals.css`와 `layout.tsx`에 심어둔, 웹을 "게임처럼" 만드는 설정들:

| 설정 | 없으면 생기는 일 |
| --- | --- |
| `h-dvh` | `100vh`는 모바일 주소창 때문에 화면이 잘림. `dvh`(dynamic viewport height)는 주소창이 늘고 줄어드는 걸 반영한 "실제 보이는 높이" |
| `overscroll-behavior-y: none` | 화면을 아래로 당기면 게임이 새로고침됨 (pull-to-refresh 참사) |
| `user-select: none` + `-webkit-touch-callout: none` | 펫을 길게 누르면 텍스트 선택/이미지 저장 메뉴가 뜸 |
| `viewport: userScalable: false` | 빠르게 연타(냠냠)하면 더블탭 줌으로 화면이 확대됨 |
| `env(safe-area-inset-*)` | 노치 있는 폰에서 버튼이 홈 인디케이터에 깔림 |

## 6. 이번 편의 게임 규칙 구현 (기획서 대조)

- 배터리 0 → **절전 모드**: 냠냠 불가(💤 잠꼬대만), 둥실 애니메이션 정지, 상태등 빨강. **사망 없음.**
- 데이터 100% → **수집 효율 50% 감소** + 전송 버튼이 빨갛게 깜빡이며 유도.
- 내구도 하한선 30%는 상수(`DURABILITY_FLOOR`)로 준비 — 파편 충돌 이벤트가 생기는 편에서 사용.
- 배터리는 5초마다 1씩 자연 소모(데모용 속도). 실제 방치 보상은 **서버 시간 기준**으로
  다음에 구현합니다 — 클라이언트 시계는 조작 가능하니 절대 믿지 않기로 했죠.
