# 개발 일지 #6 — 한 폴더에 두 사람: 브랜치, 워크트리, 협업 준비 🌿

> 2026-07-21 · 협업 워크플로 전환
> 이번 편에서 배우는 것: **브랜치의 실체(포인터), git switch의 함정, git worktree, 원격 추적 브랜치(-u), 충돌 예방 수칙**

동료가 프로젝트에 합류했습니다. 그런데 상황이 특별합니다 — **같은 저장소를 넘어, 같은 컴퓨터의 같은 폴더**에서 작업이 겹칠 수 있는 환경이에요. 실제로 이 글을 쓰는 사이에 옆 세션이 `joop-01/` 폴더에서 브랜치 `1`을 만들어 협업 규칙을 커밋했습니다. 이번 편은 그 옆에서 **브랜치 `2`를 안전하게 준비한 기록**입니다.

## 0. 팀 규칙 먼저 (CLAUDE.md에 명문화됨)

1. `main`은 공유 안정 브랜치 — **직접 커밋 금지**, 병합은 GitHub PR로.
2. 오너 세션의 작업 브랜치는 `1`, 동료(이 워크트리)는 `2`. 각자 자기 번호에서 작업합니다.
3. 작업 브랜치는 수시로 `main`을 리베이스/병합해 최신 유지.
4. 배포 버전 업(`package.json`)은 `main` 병합 시점에.

## 1. 브랜치는 복사본이 아니라 책갈피다

파이썬에서 `b = a`는 리스트를 복사하지 않습니다 — 같은 객체를 가리키는 이름이 하나 늘어날 뿐이죠. git 브랜치가 정확히 그렇습니다. **브랜치는 특정 커밋을 가리키는 40바이트짜리 포인터**일 뿐, 코드 전체의 복사본이 아닙니다. 그래서 브랜치를 100개 만들어도 저장소는 무거워지지 않습니다.

```bash
git branch 2 origin/main   # "origin/main이 가리키는 커밋에 책갈피 '2'를 꽂아라"
```

## 2. 함정: 한 폴더에는 체크아웃이 하나뿐

그럼 `git switch 2`로 갈아타면 끝일까요? 여기에 함정이 있습니다. **`git switch`는 작업 폴더의 파일들을 통째로 그 브랜치 내용으로 바꿔치기**합니다. 동료가 같은 폴더에서 브랜치 `1`의 파일을 에디터로 열어둔 채 작업 중인데 내가 `switch`를 해버리면? 동료 눈앞에서 파일이 다른 세계의 내용으로 뒤바뀝니다. 개발 서버(HMR)는 영문 모를 리로드를 반복하고요.

한 폴더 = 한 체크아웃. 이건 git의 구조적 제약입니다.

## 3. 해법: git worktree — 저장소 하나, 작업 폴더 여러 개

```bash
git worktree add ../joop-01-branch-2 -b 2 origin/main
```

이 한 줄로 `joop-01/` **옆에** 새 폴더가 생기고, 거기에 브랜치 `2`가 체크아웃됩니다. 핵심은 **`.git` 저장소(커밋·객체 데이터베이스)는 하나를 공유**하고 작업 폴더만 늘어난다는 것. 파이썬 비유로는, 하나의 데이터베이스 파일에 커서를 두 개 열어 서로 다른 뷰를 보는 것과 같습니다 — 데이터는 한 곳에, 시점만 각자.

```
JOOPs/
├── joop-01/           ← 오너 세션, 브랜치 1 (원본 .git 보유)
└── joop-01-branch-2/  ← 이 워크트리, 브랜치 2 (.git은 파일 하나로 원본을 가리킴)
```

덤으로 안전장치도 있습니다: **같은 브랜치를 두 워크트리에서 동시에 체크아웃하는 건 git이 에러로 막아줍니다.** "둘이 같은 브랜치를 건드리는 사고"가 도구 레벨에서 차단되는 거죠.

> **배울 포인트:** 워크트리는 "브랜치 전환" 문제를 "폴더 분리"로 바꿉니다.
> 리뷰하다가 내 작업으로 돌아올 때 stash를 쌓을 필요도 없어져요. 혼자 쓸 때도 유용한 도구입니다.

## 4. 워크트리가 안 가져오는 것들

새 폴더를 열어보면 뭔가 허전합니다. **git이 추적하는 파일만 체크아웃되기 때문**입니다:

- `.env.local` (gitignore 대상 — Supabase·VAPID 키) → 원본 폴더에서 **복사**
- `node_modules/` → **`npm install` 다시 실행**

파이썬으로 치면 저장소를 새로 받은 뒤 venv를 다시 만들고 `pip install -r requirements.txt` 하는 것과 같습니다. `package-lock.json`이 `uv.lock`처럼 정확한 버전을 고정해 주므로, 어느 폴더에서든 동일한 의존성 트리가 재현됩니다.

## 5. `push -u` — 원격에 내 자리 만들기

```bash
git push -u origin 2
```

`-u`(--set-upstream)는 로컬 `2`와 원격 `origin/2`를 **짝으로 등록**합니다. 이후엔 인자 없이 `git push` / `git pull`만 치면 되고, `git status`가 "원격보다 몇 커밋 앞섰는지"를 알려줍니다. 무엇보다 **동료가 GitHub에서 브랜치 `2`의 존재와 진행 상황을 볼 수 있게** 되는 게 협업 관점의 핵심입니다.

## 6. 충돌을 피하는 생활 수칙

충돌(conflict)은 "같은 파일의 같은 부분을 서로 다르게 고친 채 만났을 때"만 발생합니다. 예방이 치료보다 쉽습니다:

1. **작업 시작 전 `git fetch origin`** — 동료가 어디까지 갔는지 먼저 확인. (실제로 이번 준비 중에도 fetch 덕분에 브랜치 `1`의 등장을 바로 알았습니다.)
2. **`main`이 갱신되면 곧 리베이스** — 묵힐수록 충돌 덩치가 커집니다. `git rebase origin/main`.
3. **작고 자주 커밋, 완성 단위로 PR** — 거대한 일괄 병합이 충돌의 주범.
4. **공유 파일은 미리 조율** — 이 저장소에선 `CLAUDE.md`, `package.json`(version)이 충돌 단골. 반면 `supabase/migrations/`는 타임스탬프 파일명 덕에 자연히 충돌을 피합니다.
5. **동료의 폴더는 건드리지 않기** — 워크트리로 공간을 나눈 이유 그 자체.

> **배울 포인트:** 협업 git의 8할은 명령어가 아니라 **타이밍**입니다.
> fetch를 자주 해서 서로의 위치를 알고, 병합 단위를 작게 유지하는 습관이 도구보다 중요해요.
