# 개발 일지 #7 — 서버가 먼저 말을 거는 앱: 푸시 백엔드와 크론 ⏰🛰️

> 2026-07-21 · 브랜치 `2`의 첫 기능 · 일곱 번째 구현
> 이번 편에서 배우는 것: **풀→푸시 전환, 구독 주소록, 투영(projection) 계산, service_role 키, pg_cron 스케줄링, 멱등한 배치의 기본기**

3편에서 웹 푸시를 붙였지만 사실 반쪽짜리였습니다. 구독이 브라우저 안에만 있어서, 유저가 설정 시트를 열고 버튼을 눌러야만 "자기 자신에게" 보낼 수 있었죠. 진짜 목표는 반대 방향입니다 — **유저가 앱을 꺼 둔 사이, 서버가 펫 상태를 보고 먼저 말을 거는 것.** 배터리가 깜빡이면 "충전해 주세요", 12시간 넘게 안 오면 "보고 싶어요".

## 0. 풀(pull)에서 푸시(push)로

지금까지 줍스의 모든 정산은 **유저가 와야** 일어났습니다(4편의 `settle_offline`). 이번 편은 처음으로 **유저 없이 서버 혼자 도는 루프**를 만듭니다. 필요한 부품은 세 가지:

1. **주소록** — 누구에게 보낼 수 있나 (`joop_01_push_subscriptions`)
2. **판단** — 지금 누가 알림을 받아야 하나 (`joop_01_push_candidates()`)
3. **시계** — 주기적으로 이 판단을 실행할 스케줄러 (pg_cron)

## 1. 구독 = 주소록 한 줄

브라우저의 푸시 구독은 "이 기기로 알림을 보낼 수 있는 주소(endpoint) + 암호화 열쇠(p256dh/auth)"입니다. endpoint가 곧 기기·브라우저의 신분증이라, 같은 유저가 폰과 PC에서 구독하면 행이 두 개 생깁니다.

저장 규칙은 init 때 만든 관례를 그대로 미러링했습니다: **읽기는 본인 것만(RLS), 쓰기는 security definer RPC로만.** 재미있는 설계 하나 — 같은 endpoint가 다른 계정으로 다시 저장되면 소유자를 갈아끼웁니다. endpoint를 실제로 쥔 건 지금 이 브라우저니까, "브라우저가 곧 주인"이 맞는 규칙이거든요(익명 계정이 바뀌는 경우에 필요합니다).

수명주기도 세 군데서 챙깁니다: 켜기(저장) · 끄기(해지 **전에** endpoint를 확보해서 삭제 — 해지 후엔 읽을 수 없어요) · 그리고 **자가 치유** — 시트를 열 때마다 브라우저 구독을 DB에 조용히 재저장해서, DB 초기화나 계정 교체로 생긴 드리프트를 복구합니다. 서비스 워커의 `pushsubscriptionchange`(브라우저가 일방적으로 주소를 갈아끼우는 이벤트)도 REST 통로로 재등록합니다 — SW는 React 밖의 존재라 Server Action을 못 부르거든요.

## 2. 투영 계산 — 상태를 바꾸지 않고 "지금"을 내다보기

크론이 돌 때 DB의 `battery`는 **유저가 마지막으로 접속한 순간의 값**입니다. 진짜 현재 배터리는 이렇게 내다봅니다:

```sql
battery_now = greatest(0, battery - 소모율 × 경과시간)
```

`settle_offline`과 똑같은 수식이지만 결정적 차이가 있습니다 — **아무것도 갱신하지 않는 read-only**라는 것. 정산(쓰기)은 유저가 접속했을 때 한 번만, 알림 판단(읽기)은 크론이 15분마다 몇 번이든. 읽기와 쓰기를 분리하면 "크론이 두 번 돌면 보상이 두 번 지급되나?" 같은 걱정이 아예 사라집니다.

대신 대가가 하나 있습니다. init 마이그레이션은 협업 규칙상 손댈 수 없어서(6편 참조) 소모율 상수를 **복제**해야 했어요. 코드에 같은 진실이 두 곳에 사는 건 위험하므로, 양쪽에 "원본을 바꾸면 여기도 바꿔라" 경고 주석을 박아 두었습니다.

> **배울 포인트:** 단일 진실 원천(Single Source of Truth)을 지킬 수 없는 상황이라면,
> 차선책은 **복제를 숨기지 말고 시끄럽게 표시**하는 것입니다. 조용한 복제가 제일 무섭습니다.

## 3. service_role — RLS를 우회하는 만능 열쇠

후보 조회는 "전체 유저"를 훑어야 하는데, RLS 세상에서 유저 세션은 자기 행만 봅니다. 그래서 크론 경로는 **service_role 키**로 접속합니다 — 파이썬으로 치면 웹 요청 세션과 무관하게 admin 계정으로 여는 DB 커넥션이죠. 취급 규칙 세 가지:

1. 서버 전용 코드에서만 import (환경변수에 `NEXT_PUBLIC_` 금지 — 붙는 순간 번들에 새겨져 공개됩니다)
2. 없으면 기능이 조용히 꺼지는 폴백 (`getSupabaseAdminClient() → null`, env.ts 패턴 미러)
3. 만능 열쇠라도 쓸 수 있는 문은 좁게 — grant를 크론 전용 RPC 3종에만 줬습니다

## 4. 스케줄러 고르기 — 무료 플랜의 산수

| 후보 | 빈도 | 탈락 사유 |
|---|---|---|
| Vercel Cron 단독 | **하루 1회** (Hobby) | 배터리 20% → 절전까지 **~4.8시간**의 알림 창을 대부분 놓침 |
| Supabase Edge Function | 분 단위 | Node `web-push`를 Deno용으로 재작성해야 — 검증된 코드를 버리는 비용 |
| **pg_cron + pg_net → Next 라우트** | **15분** | ✅ 채택 — 검증된 발송 코드 재사용 + 무료 티어 OK |

pg_cron(DB 안의 스케줄러)이 pg_net(DB 안의 HTTP 클라이언트)으로 우리 라우트를 두드리는 구조입니다. 보너스: 시크릿 환경변수 이름을 `CRON_SECRET`으로 지으면 Vercel Cron이 자동으로 같은 `Authorization: Bearer` 헤더를 붙여 줘서, **하루 1회 Vercel 폴백**을 설정 한 줄로 얻습니다(pg_cron이 죽어도 최소한의 안전망).

## 5. 멱등성 — "다시 실행해도 안전한가?"라는 질문

배치 작업 설계의 제1질문입니다. 우리의 답:

- **후보 산정이 쿨다운 기반** — (유저 × 알림 종류)별 `last_notified_at`이 지나야 다시 뽑힘. 디스패처를 연속 두 번 불러도 두 번째는 빈손입니다.
- **성공만 기록** — 발송 실패는 mark하지 않으니 다음 주기가 자동 재시도. 별도 재시도 큐가 필요 없습니다.
- **410 Gone은 즉시 정리** — 푸시 서버가 "이 주소는 폐기됐다"고 알려주면 재시도 대신 구독 삭제. 죽은 주소로의 영원한 재시도를 막습니다.
- **폭주 안전장치** — 후보 `limit 500`, 놓친 분량은 다음 주기(15분 뒤)가 이어받습니다.

## 6. 검증에서 배운 것 — 가짜 데이터도 형식은 진짜여야

로컬 검증 중 재미있는 실패: 가짜 구독(`p256dh='p256dh-test'`)으로 발송을 시험했더니 **HTTP 요청도 나가기 전에** `web-push`가 거부했습니다 — "p256dh는 65바이트여야 한다". 페이로드 암호화에 진짜 타원곡선 공개키가 필요하기 때문이죠. `crypto.createECDH('prime256v1')`로 형식만 유효한 키를 만들어 넣자, 이번엔 진짜 푸시 서버가 410을 돌려줬고 prune 경로까지 검증됐습니다.

```
디스패처 응답: {"candidates":1,"sent":0,"pruned":1,"failed":0}  ← 죽은 구독 정리 확인
```

> **배울 포인트:** 외부 시스템을 흉내 낼 때는 "어느 층에서 검증이 일어나는지"를 알아야 합니다.
> 형식 검증(라이브러리) → 전송(네트워크) → 의미 검증(상대 서버), 층마다 다른 가짜가 필요해요.

## 7. 협업 관점 — 충돌 반경 밖에서 기능 만들기

이 기능 전체가 **동료(브랜치 1)가 수정 중인 파일을 하나도 건드리지 않고** 만들어졌습니다: 새 타입은 `types.ts` 대신 `push-types.ts`로, DB는 init 수정 대신 증분 마이그레이션 2개로, 로컬 검증도 공유 스택에 `db reset` 없이 새 객체만 증분 적용. 6편에서 세운 "충돌은 예방이 치료보다 쉽다"의 실전편인 셈입니다.

**남은 일(배포 시점):** 클라우드에 마이그레이션 적용 → 대시보드에서 pg_cron·pg_net 활성화 → Vercel 환경변수(`SUPABASE_SERVICE_ROLE_KEY`, `CRON_SECRET`) → `scripts/setup-push-cron.sql` 치환 실행. 상세 절차는 그 파일의 치트시트에 있습니다.
