# 개발 일지 #3 — PWA: 줍스를 '진짜 앱'으로 만든 날 📲

> 2026-07-21 · PWA 완전 구현
> 이번 편에서 배우는 것: **서비스 워커 생명주기, 캐시 전략, 배포 자동 반영, beforeinstallprompt, 웹 푸시(VAPID)와 Server Action, 그리고 웹 기술의 한계를 우아하게 다루는 법**

줍스는 앱스토어 없이 브라우저로 접속하는 모바일 웹 게임입니다. 그런데 유저 입장에서
"매번 주소 치고 들어오는 웹사이트"와 "홈 화면 아이콘 하나로 켜지는 앱"은 체감이 완전히
다르죠. 오늘은 줍스를 **PWA(Progressive Web App)** 로 만들어 그 간극을 없앴습니다.

이번에 추가된 파일들:

```
app/
  manifest.ts              ← 웹 앱 매니페스트 ("나는 설치 가능한 앱입니다" 신분증)
public/
  sw.js                    ← 서비스 워커 (캐시 프록시 + 푸시 수신기)
  offline.html             ← 통신 두절 시 보여줄 폴백 페이지
  icons/                   ← 줍이 얼굴 아이콘 세트 (일반/maskable/애플용)
lib/
  pwa.ts                   ← 버전 상수 + iOS/standalone 판별 유틸
  push-client.ts           ← 브라우저 쪽 푸시 구독 유틸
  push-messages.ts         ← 게임 기획 기반 알림 문구 템플릿
stores/
  pwa-store.ts             ← 업데이트 대기/설치 프롬프트 전역 상태
components/pwa/
  pwa-provider.tsx         ← 배선반: SW 등록 + 이벤트 포획 + UI 마운트
  update-toast.tsx         ← "새 버전 도착" 토스트
  install-nudge.tsx        ← 홈 화면 설치 넛지 (백오프 정책)
  dynamic-favicon.tsx      ← 펫 상태 따라 변하는 파비콘 + 앱 배지
components/home/
  console-settings.tsx     ← ⚙️ 관제 설정 시트 (알림/설치/버전 정보)
app/actions/
  push.ts                  ← 푸시 발송 Server Action (서버 전용)
scripts/
  generate-icons.mjs       ← 아이콘 PNG 생성기
  generate-vapid-keys.mjs  ← 푸시 서명 키 생성기
```

## 1. PWA의 최소 조건 — 신분증과 상주 데몬

브라우저가 "이 사이트는 설치해도 좋다"고 판단하는 조건은 세 가지입니다:

1. **HTTPS** — Vercel이 기본 제공
2. **웹 앱 매니페스트** — 앱 이름, 아이콘, 실행 방식을 담은 JSON
3. **서비스 워커** — 오프라인에서도 응답할 수 있는 백그라운드 스크립트

매니페스트는 `app/manifest.ts` 하나면 끝입니다. Next.js가 `/manifest.webmanifest`로
서빙하고 `<link>` 태그까지 자동으로 넣어줘요. 여기서 게임 기획과 연결되는 설정 두 가지:

- `display: "standalone"` — 설치 후 주소창 없이 전체 화면 실행
- `orientation: "portrait"` — **기획 원칙인 세로 전용 UX를 OS 레벨에서 고정**

아이콘이 4종류인 게 의아할 수 있는데, 안드로이드는 기기마다 아이콘을 원형/스쿼클로
**잘라서** 씁니다. 그래서 잘려도 안전하게 여백을 준 `maskable` 버전을 따로 만들었어요.
`scripts/generate-icons.mjs`가 홈 화면 위성 SVG와 같은 도면으로 전부 구워냅니다 —
게임 속 줍이와 홈 화면 아이콘 줍이가 똑같이 생긴 이유입니다.

## 2. 서비스 워커 — 요청을 가로채는 미들웨어 데몬

`public/sw.js`가 이번 편의 주인공입니다. 파이썬으로 비유하면 **브라우저와 네트워크
사이에 끼어 있는 미들웨어 데몬**이에요. 페이지와 별개 스레드에서 상주하면서:

- 모든 `fetch`(네트워크 요청)를 가로채 캐시로 응답하거나
- 앱이 꺼져 있어도 푸시 메시지를 받아 알림을 띄웁니다

요청 종류마다 캐시 전략을 다르게 가져갔습니다. 이 세 가지가 사실상 캐시 전략의 전부라서,
한 번 이해해 두면 어디서든 써먹습니다:

| 대상 | 전략 | 이유 |
| --- | --- | --- |
| HTML (페이지 이동) | **네트워크 우선** | 항상 최신 배포를 보여줘야 하니까. 실패하면 캐시 → 오프라인 페이지 순서로 후퇴 |
| `/_next/static/*` | **캐시 우선** | 파일명에 내용 해시가 박혀 있어서 내용이 바뀌면 이름도 바뀜 → 한 번 캐시하면 영원히 안전 |
| 아이콘 등 나머지 | **stale-while-revalidate** | 캐시로 즉시 응답하고, 뒤에서 새 버전을 받아 다음번을 위해 갱신 |

> **배울 포인트:** "전부 캐시하면 빠르겠지"는 함정입니다. HTML을 캐시 우선으로 하면
> 배포를 해도 유저가 옛날 화면을 보게 돼요. **불변인 것만 공격적으로 캐시하고, 변하는
> 것은 네트워크 우선** — 이게 철칙입니다.

## 3. 배포 자동 반영 — 이번 편의 핵심 문제

서비스 워커에는 악명 높은 함정이 있습니다. 브라우저는 등록된 SW 파일이 **바이트 단위로
같으면** 업데이트를 하지 않고, 캐시가 남아 있으면 유저는 계속 옛 버전을 봅니다.
"배포했는데 반영이 안 돼요"의 90%가 이것 때문이에요.

줍스의 해법은 **버전을 URL에 새기는 것**입니다:

```
등록 URL:  /sw.js?v=0.1.0-mrtf66ly
                    └─┬─┘ └──┬───┘
              package.json   빌드 스탬프 (빌드할 때마다 변경)
```

1. `next.config.ts`가 빌드 시작 시점에 `package.json`의 버전과 빌드 스탬프를
   `process.env.NEXT_PUBLIC_*` 자리에 **문자열 상수로 구워** 넣습니다. 실행 중에
   환경변수를 읽는 게 아니라, 파이썬으로 치면 배포 전에 템플릿 치환으로 값을 박아
   넣는 것에 가까워요.
2. 배포마다 스탬프가 달라지니 SW 등록 URL도 달라지고, 브라우저는 "다른 파일이네?"
   하며 **반드시 새 워커를 설치**합니다.
3. 캐시 이름에도 이 버전이 들어가서(`joops-0.1.0-mrtf66ly`), 새 워커가 활성화되는
   순간 이전 배포의 캐시를 전부 청소합니다.

그런데 새 워커는 바로 교대하지 않고 **대기(waiting)** 상태로 기다립니다. 기존 탭이
옛 캐시로 돌아가는 중에 새 캐시가 끼어들면 파일 버전이 뒤섞이기 때문에, 브라우저가
일부러 안전장치를 둔 거예요. 줍스는 이 순간을 UI로 만들었습니다:

```
새 워커 설치 감지 → 화면 위에 조용히 "🛰️ 새 버전이 도착했어요 [업데이트]"
[업데이트] 탭 → SW에 SKIP_WAITING 메시지 → 새 워커 교대(controllerchange) → 1회 새로고침
```

강제 새로고침하지 않는 이유: 나중에 액션 모드 플레이 중에 화면이 갑자기 리셋되면
안 되니까요. **업데이트 타이밍의 결정권은 유저에게** 줍니다. 오래 켜 둔 탭도 배포를
놓치지 않도록 30분마다 + 탭으로 돌아올 때마다 업데이트를 점검합니다.

버전 확인은 홈 화면 헤더의 `CLEAR SKY ‧ 관제 콘솔 v0.1.0` — 관제 장비 펌웨어 버전처럼
슬쩍 넣었습니다. 배포 후 이 숫자(그리고 ⚙️ 설정 시트의 빌드 스탬프)만 보면 반영 여부를
바로 알 수 있어요.

> **배울 포인트:** `controllerchange`에서 무조건 `location.reload()`를 하면 안 됩니다.
> 첫 방문 때 `clients.claim()`으로도 이 이벤트가 발생해서, 게임을 시작하자마자 화면이
> 새로고침되는 사고가 나요. "이전에 controller가 있었을 때만 리로드" 가드를 꼭 두세요
> (`pwa-provider.tsx` 참고).

## 4. 설치 넛지 — "주기적으로, 그러나 거슬리지 않게"

크롬 계열 브라우저는 설치 가능해지면 `beforeinstallprompt` 이벤트를 쏘는데, 기본 동작에
맡기면 브라우저 멋대로 미니 배너를 띄웁니다. 우리는 이걸 `preventDefault()`로 막고
이벤트 객체를 Zustand 스토어에 보관해 뒀다가 **우리가 원하는 타이밍에** 꺼내 씁니다.

넛지 정책 (`install-nudge.tsx`):

- 이미 설치했거나 앱으로 실행 중이면 절대 안 나옴
- **오프닝 컷신(첫 교감의 90초!) 중에는 절대 방해하지 않음** — 온보딩 원칙과 충돌 금지
- 접속 25초 후, 화면을 가리지 않는 하단 카드로 슬쩍 등장
- [다음에]를 누를 때마다 재등장 간격이 **3일 → 7일 → 14일로 백오프** — "주기적이지만
  귀찮게 굴수록 더 뜸해지는" 예의 바른 알림

브라우저 분기도 여기서 처리합니다. iOS Safari는 `beforeinstallprompt`가 아예 없어서
"공유 버튼(⬆️) → 홈 화면에 추가" 안내 문구를 보여주고, 데스크톱 크롬/엣지는 안드로이드와
같은 이벤트로 동작하니 같은 코드가 그대로 돌아갑니다. 넛지를 놓친 유저를 위해 ⚙️ 관제
설정 시트에도 설치 버튼을 상시 두었습니다.

> **배울 포인트:** `beforeinstallprompt`는 **한 번 쓰면 소멸하는 1회용** 이벤트입니다.
> 여러 컴포넌트(넛지, 설정 시트)가 함께 쓰려면 이벤트를 잡는 곳과 쓰는 곳을 분리해야
> 하고, 그래서 `pwa-store.ts`라는 전역 스토어가 필요했어요.

## 5. 웹 푸시 — 앱이 꺼져 있어도 줍이가 부르는 법

웹 푸시의 등장인물은 셋입니다:

```
[우리 서버] ──(VAPID 서명 + 암호화된 메시지)──▶ [브라우저 벤더의 푸시 서버] ──▶ [서비스 워커]
 Server Action                                    구글/애플이 운영                sw.js가 알림 표시
```

- **VAPID 키 쌍**: "이 푸시는 줍스 서버가 보낸 게 맞다"는 서명용 공개키/비밀키.
  `node scripts/generate-vapid-keys.mjs`로 생성해 `.env.local`(로컬)과 Vercel
  환경변수(배포)에 넣습니다. 공개키만 `NEXT_PUBLIC_` 접두사로 클라이언트에 노출합니다.
- **구독**: 유저가 관제 설정에서 [켜기]를 누르면 권한 요청 → 브라우저가 "이 키의 주인만
  나한테 보낼 수 있는 주소"(구독 엔드포인트)를 발급 → 이걸로 서버가 언제든 발송 가능.
- **발송**: `app/actions/push.ts`의 Server Action이 `web-push` 라이브러리로 쏩니다.
  `"use server"` 함수는 클라이언트에서 함수 호출하듯 부르지만 실제로는 서버에서
  실행돼요 — Flask 라우트 핸들러를 import해서 쓰는 느낌인데, HTTP 왕복은 Next.js가
  대신 처리합니다. **비밀키가 필요한 일은 반드시 이렇게 서버로** 보냅니다.

알림 문구는 게임 기획서에서 뽑아 `lib/push-messages.ts`에 템플릿으로 정리했습니다:
배터리 방전 임박 🔋, 데이터 용량 가득 📡, 오프라인 순찰 보상 ☄️, 장기 방치 시무룩 🛰️.
같은 종류의 알림이 쌓여 유저를 괴롭히지 않도록 `tag`로 덮어쓰기도 걸어뒀어요.

지금은 관제 설정의 **[테스트 전파 수신해 보기]** 버튼이 이 중 하나를 무작위로 자기
자신에게 쏘는 데모 단계입니다. Supabase가 붙으면 구독 정보를 DB에 저장하고, Edge
Function + cron이 유저별 펫 상태(`last_login_at`, 배터리 잔량)를 보고 실제 타이밍에
발송하게 됩니다 — "클라이언트 시간을 신뢰하지 않는다"는 개발 원칙 그대로요.

iOS 제약 하나: **iOS는 홈 화면에 설치된 PWA에서만 푸시가 됩니다** (iOS 16.4+).
관제 설정이 이 상황을 감지해 "홈 화면에 추가한 뒤 알림을 켤 수 있어요"라고 안내합니다.
설치 유도와 푸시가 자연스럽게 맞물리는 구조죠.

## 6. 캐릭터 동기화 아이콘 — 되는 것과 안 되는 것

"앱 아이콘이 지금 내 펫 상태에 따라 변하면 어떨까?"라는 아이디어, 결론부터 말하면:

| 표면 | 실시간 변경 | 줍스의 구현 |
| --- | --- | --- |
| 홈 화면 앱 아이콘 | ❌ OS가 설치 시점에 파일로 복사해 감 | 진화(1→3단계) 때 매니페스트 아이콘 교체로 대응 예정 |
| 브라우저 탭 파비콘 | ⭕ `<link rel="icon">` 교체 가능 | **펫 상태 실시간 반영** |
| 설치된 앱 아이콘의 뱃지 | ⭕ Badging API | **돌봐줘야 할 일 개수 표시** |

`dynamic-favicon.tsx`가 64×64 캔버스에 미니 줍이를 그려서 파비콘으로 갈아끼웁니다.
평소엔 초록 상태등에 동그란 눈, 절전 모드면 빨간 상태등에 감은 눈, 데이터가 가득 차면
인디고 알림 점이 붙어요. 데스크톱에서 다른 탭을 보다가도 파비콘만으로 줍이의 상태를
알 수 있습니다. 설치된 PWA에서는 `navigator.setAppBadge(n)`으로 아이콘에 숫자 뱃지를
달아 "지금 돌봐줘야 할 일"(절전/데이터 가득) 개수를 보여줍니다.

홈 화면 아이콘 자체는 못 바꾸는 대신, 나중에 줍이가 **진화**하면 매니페스트의 아이콘
경로를 바꿔 재배포하는 방식으로 "성장한 줍이"를 아이콘에 반영할 수 있습니다. 안드로이드는
매니페스트 변경을 주기적으로 확인해서 아이콘을 갱신해 주거든요 (즉시는 아니고 며칠 내).

> **배울 포인트:** 웹 플랫폼에는 명확한 한계선이 있습니다. 안 되는 걸 억지로 흉내 내기보다
> "되는 표면(파비콘, 뱃지)에서 최대한 표현하고, 안 되는 표면은 다른 이벤트(진화)와 묶는"
> 식으로 기획을 기술에 맞게 접어 넣는 것도 프론트엔드 개발자의 일이에요.

## 7. 직접 확인하는 법

서비스 워커는 개발 서버(HMR)와 상극이라 **프로덕션 빌드에서만** 등록됩니다:

```bash
# 1. 푸시 키 생성 (최초 1회) → 출력을 .env.local에 저장
node scripts/generate-vapid-keys.mjs

# 2. 프로덕션 빌드 + 실행
npm run build && npm start
```

- 크롬 DevTools → Application 탭 → Manifest / Service Workers에서 상태 확인
- 주소창 오른쪽에 설치 아이콘(⊕)이 뜨는지 확인
- ⚙️ 관제 설정 → 알림 켜기 → 테스트 전파 수신
- **배포 반영 테스트**: 코드를 아무거나 고치고 다시 `npm run build && npm start` →
  열려 있던 탭에 "새 버전이 도착했어요" 토스트가 뜨는지 (탭을 숨겼다 다시 보이면 즉시 점검)
- 오프라인 테스트: DevTools → Network → Offline 체크 후 새로고침

Vercel 배포 시에는 환경변수 3개(`NEXT_PUBLIC_VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`,
`VAPID_SUBJECT`)를 프로젝트 설정에 등록하는 것만 잊지 마세요.
