> 이번 편에서 배우는 것: **React Three Fiber(R3F), 씬 그래프, useFrame 게임 루프, 쿼터니언 합성, 지수 감쇠 보간, GLB 포맷과 리깅 규약**

지금까지 줍이는 2D SVG 스티커였습니다. 귀엽긴 했지만 기획서가 그리는 "로우폴리 3D 위성 펫"과는 거리가 있었죠. 오늘 `plan/img/`에 도착한 **캐릭터 에셋 팩 v2**(파이썬 trimesh로 생성된 GLB 6종!)를 받아, 줍이를 진짜 3D로 승격시켰습니다.

## 0. 에셋 팩이 이미 다 정해놨다

이번 작업이 순조로웠던 건 에셋 팩이 **계약(contract)을 문서화**해뒀기 때문입니다:

- **리깅 규약:** PascalCase 노드(`Body`, `Antenna_Tip`, `Panel_L/R`, `StatusLight`…)만 애니메이션 대상. 힌지 피벗이 노드 원점에 정렬돼 있어 **회전값만 주면** 접힘·파닥임이 된다.
- **감정 → 포즈 레시피:** 시무룩=안테나 처짐 0.9·호버 45%, 절전=처짐 1.2·날개 접힘 0.8, 동면=완전 접힘·호버 0… 우리 무드 시스템(4편)과 정확히 1:1 대응.
- **레퍼런스 뷰어:** 감정 연출이 구현된 three.js 코드가 동봉돼 있어, 수치를 검증하며 R3F로 포팅했다.

> **배울 포인트:** 아트와 코드 사이의 계약이 문서화되면 구현은 번역 작업이 됩니다.
> "이 노드를 이 축으로 돌리면 이 감정"이라는 표가 있으니 추측이 사라졌어요.

## 1. R3F — three.js를 React처럼 쓰기

three.js는 명령형입니다: `scene.add(mesh)`, `renderer.render(...)`. R3F는 이걸 React 선언형으로 감쌉니다:

```tsx
<Canvas camera={{ fov: 42, position: [0.35, 0.3, 1.6] }}>
  <hemisphereLight args={[0x9fb8ff, 0x1a2440, 0.9]} />
  <primitive object={scene} />   {/* GLB에서 로드한 씬 그래프 */}
</Canvas>
```

파이썬 비유: matplotlib의 `plt.plot()`(명령형)과 선언형 차트 라이브러리의 관계와 같습니다. JSX 태그 하나하나가 씬 그래프의 노드가 되고, React가 마운트/언마운트를 관리합니다.

**씬 그래프**는 트리 구조입니다 — `Body` 아래에 `Antenna_Base` → `Antenna_Tip`이 매달려 있어서, Base를 돌리면 Tip도 함께 돌아갑니다. 부모의 변환이 자식에게 전파되는 구조는 DOM과 똑같아서 React 개발 감각이 그대로 통합니다.

## 2. useFrame — 게임 루프의 React식 표현

매 프레임(초당 ~60회) 실행되는 훅입니다. 여기서 포즈·눈빛을 갱신합니다:

```tsx
useFrame((_, dt) => {
  const state = usePetStore.getState(); // 구독 없이 최신 상태만 읽기
  const emotion = deriveEmotion(state.mood, state.battery, state.dataUsed);
  // ... 목표 포즈로 보간, 노드 회전, 눈 emissive 갱신
});
```

중요한 설계: **여기서 setState를 하지 않습니다.** 초당 60번 리렌더가 나면 안 되니까요. 연출 상태(`droop`, `fold`, 깜빡임 타이머…)는 전부 `useRef`에 두고 three 객체를 직접 변형합니다. React는 "무대 설치"를, useFrame은 "연기"를 담당하는 분업입니다.

## 3. 프레임레이트 독립 애니메이션 — 지수 감쇠 보간

"시무룩 → 평상"으로 돌아올 때 안테나가 뚝 서면 어색합니다. 부드럽게 전환하는 고전 기법:

```ts
st.droop += (target - st.droop) * (1 - Math.exp(-4.5 * dt));
```

`현재값 += (목표 - 현재값) × 비율`인데, 비율을 `1 - e^(-k·dt)`로 쓰는 게 핵심입니다. 60fps든 30fps든 **같은 시간에 같은 거리**를 좁혀서, 저사양 폰에서도 연출 속도가 동일합니다. (그냥 `* 0.1`을 쓰면 프레임레이트에 따라 빨라졌다 느려졌다 합니다 — 방치형 게임의 "클라이언트 시계를 믿지 마라"와 닮은, "프레임 수를 믿지 마라" 원칙.)

## 4. 쿼터니언 합성 — 초기 자세를 보존하며 회전하기

처음 구현에서 만날 뻔한 함정: GLB의 힌지 노드는 **기본 자세**(matrix)를 갖고 있습니다. `node.rotation.x = droop`처럼 대입하면 기본 자세가 날아가 모델이 찌그러집니다. 해법은 로드 시점에 초기 쿼터니언을 백업하고, 매 프레임 **초기 자세 × 오프셋 회전**을 합성하는 것:

```ts
const q0 = node.quaternion.clone();           // 마운트 때 한 번
node.quaternion.copy(q0).multiply(offsetQuat); // 매 프레임
```

쿼터니언은 3D 회전의 안전한 표현법입니다(오일러 각의 짐벌락 문제 회피). "곱셈 = 회전의 합성"이라는 것만 기억해도 충분합니다.

## 5. GLB 해부와 자동 프레이밍

GLB는 `JSON(구조) + 바이너리(버텍스)` 두 청크로 된 컨테이너입니다. 이번에 Node 스크립트로 JSON 청크를 직접 파싱해 노드 계층과 메시 좌표 범위를 실측했습니다 — Panel_L이 -X 방향으로 뻗는다는 것(= 접힘 회전의 부호)도 여기서 확인했죠.

카메라 구도는 추측하지 않고 **자동 프레이밍**으로 풀었습니다:

```ts
const box = new THREE.Box3().setFromObject(scene); // 모델 전체 경계 상자
const scale = 1.15 / box.getSize(v).length최대축;   // 어떤 모델이든 같은 크기로
```

덕분에 나중에 2단계·3단계 모델로 진화해도 카메라 코드는 그대로입니다.

## 6. 제스처는 여전히 DOM이 받는다

3D가 됐지만 터치 판별은 기존 Pointer 이벤트 레이어를 그대로 씁니다 (캔버스는 `pointer-events: none`). 3D 레이캐스트로 "위성을 정확히 맞혔나"를 판정할 수도 있지만, 엄지 하나로 노는 모바일 게임에선 **펫 영역 전체가 터치 대상**인 게 UX적으로 옳습니다. 4편에서 만든 무드 시스템도, 제스처 코드도 한 줄 안 바꾸고 3D로 갈아탔습니다 — 관심사 분리(상태/제스처/렌더링)가 준 보상입니다.

## 7. 성능 (개발 원칙 3: 로우폴리만)

- 모델 63KB(4.6K 트라이), 조명 3개, 그림자 없음, `dpr` 상한 2
- 캔버스 배경 투명 → 별밤은 기존 CSS가 그대로 (GPU에 별까지 맡기지 않음)
- 눈 깜빡임·호버는 머티리얼/트랜스폼 변형뿐이라 드로우콜 증가 없음
