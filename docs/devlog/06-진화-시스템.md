## 1. 첫 증분 마이그레이션 — drop-recreate 졸업식

4편에서 만든 초기 마이그레이션은 "몇 번 돌려도 안전"하도록 `joop_01_*`를 전부
지우고 다시 만드는 방식이었습니다. 그런데 이제 클라우드에 **실제 플레이 데이터**
(오퍼레이터의 EXP 116짜리 줍이!)가 생겼으니 그 방식은 은퇴해야 합니다.

이번 마이그레이션(`20260721150000_evolution.sql`)은 예고했던 증분 방식입니다:

```sql
alter table public.joop_01_pets
  add column if not exists variant text
  check (variant is null or variant in ('net', 'magnet', 'laser'));
```

기존 행은 건드리지 않고 컬럼과 함수만 **추가**합니다. Django 마이그레이션이
`0001_initial.py` 이후 `0002_add_field.py`를 쌓아가는 것과 같은 사상 —
스키마의 역사가 파일 목록에 그대로 남습니다.

## 2. 승급도 서버가 심판한다

진화 규칙(EXP 임계값)을 클라이언트에 두면 콘솔에서 `level = 3` 한 줄로 끝입니다.
그래서 4편의 원칙을 그대로 연장했습니다 — **쓰기는 검증 로직이 있는 DB 함수로만**:

```sql
create function joop_01_evolve_pet(p_variant text default null) ...
  if v_pet.level = 1 then
    if v_pet.exp < 300 then return {'ok': false, 'reason': 'exp_low'} ...
```

- `sync_pet()`은 level을 아예 건드리지 않고, `pets`엔 update 권한이 없으므로
  승급 통로는 이 함수 하나뿐입니다.
- `for update` 행 잠금으로 더블 탭 이중 진화도 차단.
- 3단계 장비는 필수 + 화이트리스트 검증 — "hammer" 같은 건 거절됩니다.
- 실측 검증 7종: exp 부족 → 승급 → 재부족 → 장비 미선택 → 엉뚱한 장비 →
  laser 진화 → 만렙 거절. 전부 기대값 일치.

클라이언트의 `EVOLVE_EXP` 상수는 **UI 표시용 미러**일 뿐입니다. 파이썬으로 치면
프론트 폼의 `maxlength`와 서버 밸리데이터의 관계 — 진실은 언제나 서버 쪽입니다.

## 3. 진화하면 몸이 바뀐다 — 동적 GLB 스왑

5편에서 만든 3D 파이프라인 덕을 크게 봤습니다:

```ts
function modelUrlFor(level, variant) {
  if (level >= 3 && variant) return `/models/pet_stage3_${variant}.glb`;
  if (level >= 2) return "/models/pet_stage2_junior.glb";
  return "/models/pet_stage1_baby.glb";
}
// <PetModel key={url} url={url} />  ← key가 바뀌면 React가 리마운트
```

`key={url}` 한 줄이 핵심입니다. React는 key가 바뀌면 컴포넌트를 새로 만드는데,
그 덕에 리그(노드 수집·초기 자세 백업)가 새 모델 기준으로 깨끗하게 다시 준비됩니다.

상위 단계 모델은 노드 구성이 다릅니다 — 3단계는 안테나가 좌우 쫑긋 귀 2쌍
(`Antenna_L_Tip`/`Antenna_R_Tip`), 2단계부터는 아코디언 아우터 패널
(`Panel_*_Outer`, 접힘 역방향 배율 -2.9). 리그를 "이름 규약으로 있는 것만
수집"하는 방식으로 일반화해서, 어떤 모델이 와도 같은 감정 연출 코드가 돕니다.
GLB 4종의 노드 이름을 바이너리 파싱으로 실측 확인까지 마쳤습니다.
장비 마이크로 아이들(그물 후프 회전, 레이저 터렛 두리번)도 이때 함께 붙였고,
자동 프레이밍(Box3) 덕분에 카메라 코드는 한 줄도 안 바꿨습니다.

## 4. 로컬 모드 폴백 — configured라는 신호

Supabase 키가 없는 로컬 모드에서도 진화는 되어야 합니다. Server Action이
"설정 안 됨"을 명시적으로 알려주고, 클라이언트가 같은 규칙의 로컬 폴백을 씁니다:

```ts
const result = await evolvePet(variant);
if (!result.configured) usePetStore.getState().evolveLocal(variant); // 폴백
else if (result.ok) hydrateFromServer(result.pet, { keepMood: true }); // 서버가 진실
```

> **배울 포인트:** "실패"와 "기능 꺼짐"을 같은 에러로 뭉개지 않기.
> `configured: false`라는 별도 신호 덕에 호출부가 폴백/에러를 정확히 분기합니다.

## 5. UI — 승급 버튼과 평생 직업 선택

- 헤더가 실데이터로: `줍이 · LV.2`, `EXP 116 / 300` (다음 목표 표시)
- 임계값에 닿으면 상태 메시지 아래 **진화 버튼**이 맥동하며 등장
- LV.3 진화 시트는 장비 3종 카드에서 선택 — "한 번 고르면 바꿀 수 없어요.
  줍이의 평생 직업이랍니다" (물리 용어 대신 친숙한 소개, 개발 원칙 9)
- 승급 순간 화면 전체 골드 플래시 + "⬆️ LV.2 진화!" — 레벨 변화 감지는
  `prevLevelRef` 비교라서 서버 승급·로컬 폴백 어느 경로든 동작합니다
